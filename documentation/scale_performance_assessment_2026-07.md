# InfiniBot Scale & Performance Assessment (July 2026)

**Disclaimer:** This assessment was generated by Claude Fable 5 and edited by myself.

**Context:** InfiniBot runs on ~4,000 guilds / ~500k users (nextcord 3.1.1, Python 3.13,
SQLite, single VPS). Symptoms: hours-long boot, prod-only timeouts with no logs, VPS
OOM-crash after a few days, and missing Components V2 support.

**Headline conclusion:** All of these are architectural issues in the current codebase,
not limitations of Python or nextcord-class libraries. A Rust rewrite or custom library
would inherit every one of them if the same patterns were ported. Fix in place first.

---

## 1. Hours-long bootup — startup guild chunking

**Root cause:** `src/core/bot.py` creates the bot with `chunk_guilds_at_startup=True`
(and the members intent), which requests the full member list of every guild over the
gateway before `on_ready` — ~500k members across 4,000 guilds, rate-limited per shard.
`ensure_guilds_ready()` then retries stragglers with a REST fallback
(`guild.fetch_members(limit=None)`, 1,000 members/request, heavily rate-limited).
The boot time *is* the chunking.

### What actually changes with `chunk_guilds_at_startup=False`

Verified directly against nextcord 3.1.1 source (`nextcord/state.py`):

| Behavior | With chunking off |
| --- | --- |
| `on_member_join` | ✅ Unaffected — gateway payload contains the full member |
| `on_raw_member_remove` | ✅ Always dispatched, payload includes full user object (`state.py:1733`) |
| `on_member_remove` | ⚠️ **Only fires for cached members** (`state.py:1713` gates on `guild.get_member`) |
| `on_member_update` | ⚠️ First event per uncached member is skipped, but nextcord then caches the member so later updates fire (`state.py:1747`) |
| Message authors / interaction users | ✅ Built from event payloads, not the cache |
| `guild.member_count` | ✅ Maintained from GUILD_CREATE + join/remove events |
| `guild.members` / `get_member` | ⚠️ Cache is partial (see below) |

Even without chunking, `MemberCacheFlags.from_intents` keeps caching members who join,
who are in voice, and anyone seen in a `GUILD_MEMBER_UPDATE` — the cache warms itself
with *active* members. Additionally, `utils.get_member()`
(`src/components/utils.py:761`) already falls back to REST `fetch_member` on a cache
miss, so birthdays, leveling, strikes, and dashboard lookups already tolerate an
unchunked cache.

### Required code changes when disabling

1. Move `on_member_remove` logic (leave messages, level cleanup, removal logging) to
   `on_raw_member_remove` — same pattern already used for raw message edit/delete.
2. Remove the runtime `await guild.chunk()` calls in `on_raw_message_edit` /
   `on_raw_message_delete` (`src/core/bot.py:586,653`), or the bot slowly re-chunks
   everything at runtime anyway.
3. Accept (or work around) the missed *first* nickname-change event per uncached member
   in the nickname profanity check.

**Note on "store chunk data in the DB and restore on boot":** stale by construction —
members join/leave while the bot is offline, and staleness hits exactly the paths that
need correctness. Raw events + REST-on-miss make it unnecessary. The industrial version
of the idea (standalone gateway proxy + Redis cache, e.g. twilight-gateway) exists but
is overkill at this scale.

**Expected result:** boot drops from hours to well under a minute; baseline RSS drops
by GBs (no ~500k cached `Member` objects).

## 2. Prod-only failures / timeouts — synchronous data layer blocking the event loop

The bot is a single-threaded asyncio program. Any non-`await`ed I/O freezes the entire
loop — heartbeats, interaction responses, all event processing — until it returns.
The data layer is entirely synchronous and is invoked, heavily, on the per-message hot
path:

- `feature_is_active()` → `get_global_kill_status()` constructs a **new**
  `GlobalKillStatus()` on every call (`src/config/global_settings.py:291`). The
  constructor's `sync_variables_with_image()` runs ~25 `in` checks, and every
  `JSONFile.__contains__`/`__getitem__` call **re-opens and re-parses the JSON file
  from disk** (`src/config/file_manager.py:102` — `_get_data()` has no cache, and
  `ensure_existence()` parses the file an extra time per read). One call ≈ 50 file
  opens + JSON parses. `get_configs()` has the same pattern.
- `on_message` calls these several times (logging check, moderation, spam, leveling),
  plus a synchronous SQLite `INSERT` per message (`store_message_in_db`), plus `Server`
  profile properties that each issue a separate `SELECT`, plus a `does_entry_exist`
  query in every `Simple_TableManager.__init__` (`src/core/db_manager.py:147`).

**Net effect:** one message can trigger 100+ disk opens/JSON parses and several SQLite
queries, all blocking the loop. Dev traffic never notices; prod volume stalls the loop
long enough to delay gateway heartbeats (silent shard disconnect/resume) and miss the
3-second interaction window. Nothing errors — everything just runs late — hence
"timeouts on prod with no logs."

The 15-minute scheduler (`src/core/scheduling.py`) iterates all guilds doing the same
synchronous per-guild reads on the loop, compounding the stalls.

## 3. Memory leak — non-expiring views and modals in nextcord's stores

Verified against nextcord source:

- **Views:** `ViewStore._views` (`nextcord/ui/view.py:496`) holds strong references,
  evicted only when `view.is_finished()` (`__verify_integrity`, `view.py:512`). A view
  finishes on timeout or explicit `.stop()`. With `timeout=None` no timeout task is
  ever created → **the view is immortal**, along with its closures, embeds, and
  interaction/guild references.
- **Modals:** identical, plus two aggravators (`nextcord/ui/modal.py`): the store key
  uses a **random custom_id per instance** (every open = new dict entry), and
  **submitting a modal does not stop it** — `Modal._scheduled_task` (`modal.py:259`)
  just runs the callback.

The codebase constructs **~119 views/modals with `timeout=None`**, most per-interaction
(dashboard, profile, onboarding, jokes), and calls `.stop()` only ~20 times. Every
dashboard/profile/onboarding/joke interaction across 4,000 servers leaks permanently.
Days-until-OOM matches this signature exactly.

**Fix:** finite timeouts (10–15 min) on all per-interaction views/modals; `self.stop()`
in terminal callbacks; `timeout=None` only for the 10 persistent views registered once
in `init_views()` (`src/core/view_manager.py`).

**Verification:** log `len(bot._connection._view_store._views)` (e.g. from an admin
command) and watch it grow under load before the fix, plateau after.

### Swept and (mostly) clean

`tz_cache` (bounded TTLCache), `ExpiringSet` (purges on access), message cache
(stale-channel cleanup), `application_emoji_cache` (bounded). Minor items:
`channels_purging` entries orphan if a purge raises before cleanup; `on_ready` re-runs
its whole body on reconnect (duplicate startup notifications, re-init of views — not a
leak, but unwanted re-entrancy). The chunked member cache is not a leak but is the
multi-GB baseline that makes the leak fatal sooner.

## 4. Components V2 — nextcord status and migration options

- **nextcord:** 3.1.1 (installed) has no CV2. Latest release 3.2.0 (May 2026) added
  only **low-level** CV2 (raw payloads) — no `ui.LayoutView`-style framework; the
  high-level work is an open PR.
- **discord.py:** 2.6 (mid-2025) shipped the complete high-level system —
  `ui.LayoutView`, `Container`, `TextDisplay`, `Section`, `MediaGallery`, `Separator`.
  Constraints: max 40 components per LayoutView; CV2 messages cannot carry
  `content`/`embeds`. discord.py is the upstream nextcord forked from and the most
  battle-tested Python library at scale.

### Measured migration surface (nextcord → discord.py)

45 files import nextcord, ~2,000 `nextcord.*` references, but the dominant symbols
(`Embed` ×367, `ui.Button` ×294, `Color.*`, `ButtonStyle.*`, `Member`, `Guild`,
`Message`, `TextInput` ×63, `SelectOption` ×63, `errors.Forbidden`) are name-identical
in discord.py → a mechanical rename covers ~90%. Real work:

- 37 slash/user/message command registrations → `app_commands.CommandTree`
  (`/view`, `/set`, `/create` parents become `app_commands.Group`s)
- 10 `SlashOption`s → typed parameters + `app_commands.describe`
- ~10 modals rename `callback` → `on_submit`
- `contexts=[...]` → `@app_commands.guild_only()`; explicit `tree.sync()` required
- `nextcord-ext-menus` is declared in pyproject but never imported — drop it
- No nextcord-only helpers in use (zero `interaction.send()` calls)

**Estimate:** 2–4 days of focused work + ~a week of dev soak testing.

Key references: nextcord changelog (docs.nextcord.dev/en/stable/whats_new.html),
discord.py changelog (discordpy.readthedocs.io/en/latest/whats_new.html), discord.py
PR #10166, Umbra's CV2 guide
(about.abstractumbra.dev/discord.py/2025/08/17/components-v2.html).

## 5. Other bugs noticed

- `src/main.py:40` — `os.environ.get(var[0]).lower()` raises `AttributeError` when a
  var is missing (the missing-var branch above only warns, doesn't exit).
- `src/core/bot.py:503` — error handler mutates the shared global
  `INFINIBOT_ERROR_EMBED` (`set_footer`); concurrent errors cross-contaminate error IDs.
- `src/modules/database.py:105` — new `scoped_session` per query, never `.remove()`d;
  `raise Exception(e)` erases the original exception type.
- Table/column names f-string-interpolated into SQL throughout `db_manager.py` —
  internal-only today, but fragile.
- `on_ready` re-entrancy on reconnects (see §3).

## 6. Empirical validation — synthetic event harness (2026-07-04)

The claims in §2 were validated with a synthetic event harness
(`src/event_harness.py` + `run_harness.bash`, PR #70; see
`documentation/event_harness.md`). It fires fabricated Discord events directly through
the real handlers — no gateway — and measures per-event-type latency and asyncio
event-loop lag. A `--db-path` mode runs against a WAL-safe copy of a production DB
snapshot, so events hit real guild IDs and real feature profiles, with the container
network-isolated (`--network none` + stubbed `fetch_channel`) so nothing can reach the
live Discord API. Exact figures live in the harness run logs and are deliberately kept
out of this document.

Findings from benchmarking against a production snapshot:

- **§2 confirmed.** Median `on_message` handler time is in the *tens of milliseconds*,
  and it is almost entirely synchronous work on the event loop (repeated JSON config
  re-parsing plus per-property SQL SELECTs). That puts a hard ceiling of a few dozen
  messages/second on the single event loop, and means a modest burst of messages stalls
  heartbeats and interaction responses for a large fraction of a second.
- **The loop-lag signature matches the "prod-only timeouts with no logs" theory.** The
  typical watchdog tick shows sub-millisecond lag, but rare stalls reach multiple
  seconds. Dev traffic never surfaces these; prod volume hits them constantly.
- **New finding: the expensive path is the common case.** The majority of sampled
  production guilds have action logging enabled — noticeably more than the synthetic
  benchmark's assumption (a quarter of guilds) — so real hot-path costs are *higher*
  than the synthetic numbers suggested.
- Known caveats when reading harness output (documented in `event_harness.md`): the
  harness dispatches sequentially, so its throughput figure is not a prod-capacity
  number; and the `on_member_join` ~1 s tail in snapshot runs is a harness artefact —
  read the median instead.

The harness is now the regression metric for the plan below: run it (same seed, same
snapshot) before and after each change.

## 7. Prioritized remediation plan

| # | Change | Effort | Fixes | Status |
| --- | --- | --- | --- | --- |
| 0 | Synthetic event harness + prod-snapshot benchmarking (§6) | — | Baseline & regression metric for everything below | ✅ Done (PR #70) |
| 1 | In-memory cache for `JSONFile` config reads (invalidate on write; `GlobalSetting` API unchanged) | ~1 day | ~99% of hot-path file I/O; most prod flakiness | Next up |
| 2 | Fix view/modal leak (finite timeouts, `stop()` in terminal callbacks) | ~1–2 days | OOM crash | Not started |
| 3 | Disable startup chunking + raw-event migration (§1) | ~1–2 days | Boot time; GBs of RSS | Not started |
| 4 | Per-guild settings cache: one `SELECT *` per table per guild in a `TTLCache(maxsize≈5000)`, invalidated on write | ~2–3 days | Remaining hot-path queries | Not started |
| 5 | Wrap residual DB calls in `asyncio.to_thread` (or aiosqlite) | ~1 day | Belt-and-suspenders loop protection | Not started |
| 6 | Event-loop lag watchdog (task that sleeps 1s, logs when it wakes late) | ~0.5 day | Makes future blocking regressions visible | Partial — the harness ships an offline watchdog; the in-prod one is still to do |
| 7 | discord.py migration (own changeset, after the above) | ~1 wk incl. soak | Components V2, maintained upstream | Not started |

### Next steps

1. Item 1 (config cache) — smallest diff, expected to remove the bulk of the per-message
   synchronous file I/O on its own.
2. Re-run the harness benchmark (same seed, same prod snapshot) and compare the
   `on_message` median and loop-lag stats against the §6 baseline.
3. Continue down the table in order; re-benchmark after each hot-path change. Items 2
   and 3 (leak, chunking) don't show up in harness numbers — verify those with the view-
   store counter (§3) and boot-time/RSS measurements respectively.

On the radar, not blocking: Postgres only if ever multi-process (SQLite+WAL is fine
single-process at this scale); consolidate the three overlapping message caches
(nextcord `max_messages=1000` — global, not per-channel — + in-memory cache + DB
store); gateway/worker split only well past ~20k guilds.

**Rewrite verdict:** a Rust rewrite or custom library is not justified by any observed
symptom; every root cause above ports across languages if the patterns port. Revisit
only if CPU-bound after items 1–6, and then prefer a gateway/worker split over a full
rewrite.
